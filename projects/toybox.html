1

CSV 기반 데이터 로딩 구조 설계

기획-개발 간 협업과 빠른 밸런싱을 위한 범용 CSV 파싱 파이프라인

설계 배경

ScriptableObject 방식의 한계
  • Unity 에디터로만 수정 가능 — 기획자 직접 수정 불가
  • 데이터 수 증가 시 파일 탐색 비용 증가
  • 기획-개발 간 버전 충돌 빈번 발생
  • 생존 요소(허기, 목마름, 체온 등) 빠른 추가/변동에 대응 어려움
CSV 파이프라인
  • 스프레드시트에서 직접 편집 가능 → 기획-개발 분리
  • 연관 데이터를 하나의 CSV 파일로 묶어 한눈에 확인
  • DataManager 싱글톤이 게임 시작 시 단 1회 파싱 후 재사용
  • partial 클래스로 파일 단위 분리 → 확장 시 기존 코드 무영향

파이프라인 구조

CsvParser ├─ Parse() → CSV 전체를 List<string[]> (Rows) 로 반환 └─ ParseSections() → 섹션별 Dictionary<string, List<string[]>> 반환 (빈 행을 기준으로 섹션 분리) CsvRow ← 한 행을 다루는 클래스 └─ row["컬럼명"] 로 접근 / GetInt(), GetFloat(), GetBool() 제공 ICsvRecord ← CSV 한 줄 → C# 객체 변환 인터페이스 └─ FromCsvRow(CsvRow row) 구현 필수 CsvTable<T> ← Rows → Dictionary<key, T> 변환 └─ FindByKey(key) → O(1) 조회 DataManager (싱글톤, partial 클래스로 분리) └─ Init() → InitSurvivalStatDatas(), InitPlayerFixedDatas(), ... 순차 호출

데이터 추가 시 작업 순서

Step 1
CSV 파일 작성 후 Resources/Data 경로에 저장
Step 2
ICsvRecord 구현 데이터 클래스 생성 — FromCsvRow에서 컬럼 매핑
Step 3
DataManager partial 클래스에 CsvTable<T> 필드 + 초기화 함수 추가
Step 4
DataManager.Init()에 초기화 함수 한 줄 추가 → 완료

2

플레이어 시스템

MVC 패턴 + HFSM 구조를 결합해 이동·전투·생존·입력이 서로 독립적으로 동작하는 확장 가능한 플레이어 구조

전체 구조도

Player ├─ Model (데이터) │ ├─ PlayerLocomotionModel 이동 관련 스탯 │ ├─ PlayerCombatModel 공격/피격/경직, 장착 무기 데이터 │ └─ PlayerSurvivalModel 체력, 허기 등 생존 수치 (UnityEvent 옵저버 패턴) │ ├─ Controller (입력·상태 제어) │ ├─ PlayerStateMachine (HFSM) │ ├─ PlayerInputReader InputSystem 이벤트 수신 → 프레임 단위 저장 │ ├─ PlayerInputData 입력 Flag 소비 구조 (LateUpdate에서 초기화) │ └─ ColliderController 콜라이더 상태 제어 │ ├─ View (애니메이션·이벤트) │ └─ Hitbox / HitInfo (전투 판정)

Model 세분화 배경

문제
  • 단일 Model 클래스에 이동·공격·생존 데이터 집중 → 응집도 저하
  • Controller가 입력 처리 + 상태 전환을 모두 담당 → 코드 비대화
  • 상태 전환 규칙이 각 상태에 분산 → 신규 상태 추가 시 충돌
해결
  • Model을 Locomotion·Combat·Survival 도메인 단위로 분리
  • Controller 내 입력 처리(Reader/Data)와 상태 제어(HFSM) 분리
  • 상태 전환 규칙을 StateMachine 중앙에서 관리

Unity Input System 버퍼링 구조

문제: 이벤트 기반 vs 프레임 기반 충돌
  • InputSystem 콜백에서 로직 직접 호출 시 타이밍 불일치
  • "피격 중 점프 금지" 등 규칙이 행동 로직 내부에 분산
  • 동시 입력 우선순위 처리가 기능마다 복잡해짐
해결: 프레임 단위 입력 버퍼링
  • PlayerInputReader: 콜백에서 "이번 프레임에 이런 입력 있었음"만 기록
  • PlayerInputData: LateUpdate에서 모든 Flag 초기화 (1프레임 1회 보장)
  • 상태 머신이 InputSystem에 직접 의존 없이 InputData만 참조

HFSM 계층 구조

Root └─ Alive ├─ Grounded │ ├─ Idle │ ├─ Move (Walk / Run / Crouch) │ ├─ Attack (무기별 고유 모션, 점프 공격, 원거리) │ ├─ Roll (구르기) │ └─ Interact (상호작용) ├─ Airborne │ └─ Jump ├─ Stagger (경직) └─ Dead (사망)

공통 로직을 지닌 상태를 계층으로 묶어 중복 코드 방지. 계층 적용 후 Animator Transition 수가 대폭 감소하여 관리 용이성 확보.

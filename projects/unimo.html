원격 리소스 패치 시스템

Unity Addressables + Google Cloud Storage를 연동한 모바일 최적화 원격 패치 구조

설계 배경

문제
  • 기존 Resources 구조는 작은 수정에도 전체 빌드가 필요
  • 리소스가 늘어날수록 빌드 시간 증가
  • 모바일 환경에 대응하기 부적합
해결
  • Unity Addressable Asset System 적용
  • Google Cloud Storage(GCS)를 원격 카탈로그 저장소로 사용
  • 앱 실행 시 필요한 리소스만 원격 다운로드

구조

Remote 프로필
실제 출시 버전용. GCS 버킷 경로에 AssetBundle·catalog.json 업로드하여 원격 관리
InternalTest 프로필
내부 테스트 빌드용. 별도 경로로 운영하여 출시 버전과 분리
GCS 폴더 구조
버전별 폴더로 분리 관리. 임시 저장소(Temp) 운영으로 스테이징 지원

결과

⚡

전체 빌드 시간 단축리소스 변경 시 전체 빌드 없이 해당 번들만 갱신 가능

📱

어플리케이션 용량 최소화Addressable 적용 전후 실행 파일 용량 대폭 감소

🔄

유저 편의성 확대앱 버전 업 빈도 감축 — 리소스 수정 시 앱 재설치 불필요

회고

패치 후 롤백이 필요한 경우를 고려해, 이전 버전 리소스를 제거하는 방식보다 버전별 폴더링으로 빌드 버전과 산출물 버전을 일치시키며 관리하는 것이 더 안전했을 것 같다.

2

오브젝트 풀링 구조 설계

프리팹 단위 지연(Lazy) 생성 풀로 재료·SFX·VFX까지 확장 가능한 범용 풀링 구조

설계 배경

문제
  • 재료 오브젝트의 빈번한 생성/파괴 → GC 부하
  • 재료 종류 증가로 공통 프리팹 방식 비효율화
  • 모양·크기·애니메이션·데이터가 재료별로 상이
해결
  • 프리팹별 독립 풀을 동적으로 생성하는 Lazy 구조
  • 처음부터 모든 풀을 만드는 방식 대신 사용 시점에 생성
  • SFX/VFX까지 동일 구조로 확장 가능하도록 설계

구성 요소

PoolManager (싱글톤) └─ Dictionary<string, PoolBundle> (프리팹 이름 → 풀) ├─ GetPoolBundle(prefab) // 없으면 자동 생성 └─ 씬 전환 시 전체 풀 초기화 PoolBundle └─ ObjectPool (Stack<GameObject> 기반) ├─ DisposePooledObj() // 꺼내기 (없으면 Instantiate) └─ ReturnPooledObj() // 반환 후 비활성화 PooledObject (풀링 오브젝트에 부착) └─ ParentPool.ReturnPooledObj(this) // 자신을 풀로 반환

주요 특징

지연(Lazy) 생성
스테이지 진입 시 건물의 생산 재료 풀만 생성. 초기 로딩 비용 최소화
자동 확장
풀이 비었을 경우 자동으로 Instantiate. 고정 크기의 한계 없음
씬 전환 대응
씬 로드 시 기존 풀 전체 초기화하여 메모리 누수 방지
단순 접근
PoolManager.Instance.GetPoolBundle(prefab)으로 어디서든 접근 가능

결과

🧠

메모리 낭비 감소초기 로딩 비용을 최소화하고 미사용 풀을 생성하지 않음

📦

프리팹 단위 관리재료별 독립 풀로 관리 복잡도를 낮추고 범용성 확보

🔊

확장성재료뿐 아니라 SFX·VFX까지 동일 구조로 확장 적용

3

스테이지 시스템

씬 중심이 아닌 데이터 중심으로 스테이지를 관리하는 단일 씬 구조

설계 배경

문제
  • 기존 방식: 스테이지마다 독립 씬 구성
  • 스테이지 초기화 로직 수정 시 각 씬마다 변경 필요
  • 씬 수 증가에 따른 유지보수 비용 급증
해결
  • 단일 StageScene에서 모든 스테이지 공통 로직 담당
  • Firebase의 StageID를 참조해 맵·NPC·퀘스트 동적 초기화
  • StageID 하나로 전체 상태 복원 가능

초기화 시퀀스

[게임 시작] ↓ StageManager.Awake() → WaitAndInit() ├─ Firebase 초기화 대기 (WaitUntil) ├─ UserData, CurStageData, Npc, QuestList 순차 로드 ↓ Init() ├─ Addressables로 맵 씬(MapScene_StageID) Additive 로드 ├─ NPC 및 퀘스트 데이터 주입 ├─ CurrentQuestID 복원 (미완료 첫 번째 퀘스트) ├─ Camera / HUD 초기화 └─ 오프라인 보상 처리 (StageExitTimeData 비교) ↓ StageScene 플레이 시작

결과

🏗️

스테이지 구조 통합단일 씬에서 데이터 기반으로 모든 스테이지 환경을 동적 주입

➕

콘텐츠 확장성 확보신규 스테이지 추가 시 StageID와 데이터만 정의하면 코드 수정 없이 자동 대응

🔧

유지보수성 향상모든 스테이지가 동일 로직으로 초기화 — 단일 씬으로 전체 스테이지 로직 검증 가능

☁️

데이터 복원 안정성모든 진행 상태가 Firebase에 저장되어 앱 재시작 후에도 안전한 상태 복원
